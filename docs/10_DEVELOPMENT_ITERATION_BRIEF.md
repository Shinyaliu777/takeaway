# 单店外卖小程序开发迭代需求报告

更新时间：2026-03-22

## 1. 文档目的

本文档面向开发，目标不是再次讨论产品方向，而是明确下一阶段应该怎么迭代、先做什么、做到什么算完成。

当前判断：

- 项目主业务链已可用
- 当前最主要问题不在后端能力缺失，而在用户主链路体验偏重
- 下一阶段应优先优化转化、理解成本、支付信任感和商家处理效率

本文档默认以当前代码和现行 Markdown 文档为准。

## 2. 当前产品问题总结

结合当前实现，用户端和商家端的核心问题如下：

- 用户进入即强登录，首单转化门槛偏高
- 菜单页对“自动拆套餐”解释不够稳定，用户容易怀疑价格
- 购物车页已接近核对页，但信息层级还不够像结算单
- 支付截图审核流程天然偏重，需要更强的状态反馈和信任提示
- 商家首页已有工作台雏形，但还需要更明确的待办优先级
- 商品管理和规则管理可用，但缺少提升运营效率的工具

结论：

- 本轮不要优先扩新业务
- 本轮优先优化现有主链路

## 3. 本轮迭代目标

本轮开发目标分为四项：

1. 降低首次进入的流失率
2. 降低用户对套餐结算逻辑的困惑
3. 提高付款截图审核流程的透明度和信任感
4. 提高商家端日常操作效率

## 4. 迭代范围

本轮范围包含：

- 用户端首页 / 登录 / 购物车 / 订单详情
- 商家端首页 / 商品管理 / 规则管理
- 少量后端接口补充与字段补充
- 文案、状态提示、页面信息层级优化

本轮不做：

- 接入第三方在线支付
- 重构鉴权体系
- 引入新的前端框架
- 大规模重构数据库模型
- 引入完整 migration 系统

## 5. 需求优先级

### P0

- 游客可浏览菜单
- 自动拆套餐反馈增强
- 订单详情时间线增强
- 商家首页待办优先级增强

### P1

- 登录成功反馈增强
- 购物车页结算单化
- 商品管理筛选能力增强
- 规则改动效果预览增强

### P2

- 联系商家入口
- 商品复制 / 批量上下架
- 用户价值标签增强

## 6. 详细需求

## 6.1 用户端：允许游客浏览菜单

### 目标

降低首次进入时的阻断感，让用户先看到菜品、价格和店铺信息，再决定是否登录下单。

### 当前问题

当前用户端 [app.js](/Users/liuxu/Desktop/codex/takeaway-app/miniprogram-user/app.js) 和 [index.js](/Users/liuxu/Desktop/codex/takeaway-app/miniprogram-user/pages/index/index.js) 在进入菜单页前就要求登录，会直接阻断首次浏览。

### 需求说明

- 用户未登录时，也可以进入菜单页
- 用户未登录时，可以查看：
  - 店铺信息
  - 分类
  - 商品列表
  - 推荐位
  - 当前套餐规则摘要
- 用户未登录时，不可以执行：
  - 加入购物车
  - 进入购物车下单
  - 地址管理
  - 上传付款截图
- 当未登录用户点击加菜、去结算、地址等动作时，再跳转登录页

### 开发建议

- 调整 [index.js](/Users/liuxu/Desktop/codex/takeaway-app/miniprogram-user/pages/index/index.js) 的 `onShow`
- 取消菜单页首屏强制 `requireUserLogin`
- 保留需要身份的动作级拦截
- 检查 [cart.js](/Users/liuxu/Desktop/codex/takeaway-app/miniprogram-user/pages/cart/cart.js)、[order-detail.js](/Users/liuxu/Desktop/codex/takeaway-app/miniprogram-user/pages/order-detail/order-detail.js) 的登录跳转逻辑，保证一致

### 验收标准

- 未登录也能正常打开菜单页并加载菜品
- 未登录用户点击加菜时会被引导去登录
- 已登录用户主流程不受影响

## 6.2 用户端：强化套餐自动拆分反馈

### 目标

让用户明确理解：

- 当前为什么能结算或不能结算
- 系统是怎么计算价格的
- 还差什么才能下单

### 当前问题

当前前端已有 [pricing.js](/Users/liuxu/Desktop/codex/takeaway-app/miniprogram-user/utils/pricing.js) 预演逻辑，但页面表达仍不够强，产品特色没有被充分解释成“用户优势”。

### 需求说明

- 菜单页底部结算栏持续显示：
  - 已选菜品数量
  - 当前匹配到的套餐数量
  - 当前合计金额
- 当组合不完整时，显示明确缺口提示，例如：
  - `当前还差 1 个素菜才能组成套餐`
  - `当前荤菜不能单独结算，请继续补齐搭配`
- 当组合完整时，显示明确正反馈，例如：
  - `已自动匹配 2 组套餐，可直接去核对下单`
- 购物车页需要清楚展示每个套餐包含的具体菜品，不只显示总价

### 开发建议

- 优先复用 [pricing.js](/Users/liuxu/Desktop/codex/takeaway-app/miniprogram-user/utils/pricing.js) 已有的 `summaryText`、`missingHint`、`comboLines`
- 菜单页只增强展示，不重新发明一套计算逻辑
- 保证前端提示和后端 [routes.py](/Users/liuxu/Desktop/codex/takeaway-app/backend/app/api/routes.py) 的结算规则表达一致

### 验收标准

- 用户在菜单页能实时看到缺口提示或可结算提示
- 用户在购物车页能看懂每个套餐内包含哪些菜
- 前端提示和后端可结算结果一致

## 6.3 用户端：购物车页改造成标准核对页

### 目标

让购物车页承担“确认下单”而不是“重新思考规则”的职责。

### 需求说明

购物车页固定结构调整为：

1. 已选菜品
2. 套餐拆分结果
3. 地址信息
4. 付款方式说明
5. 下单按钮

页面不再强调复杂说明，不增加重复解释卡片。

### 开发建议

- 以 [cart.js](/Users/liuxu/Desktop/codex/takeaway-app/miniprogram-user/pages/cart/cart.js) 为主改造
- 优先优化信息层级和区块顺序
- 保持当前下单接口不变

### 验收标准

- 用户进入购物车页后可以快速完成核对并下单
- 页面主要信息按结算顺序组织
- 不引入新的业务分支

## 6.4 用户端：增强付款截图流程透明度

### 目标

降低用户对“截图是否提交成功、商家是否已处理、多久能确认”的不安。

### 当前问题

当前订单详情页已有时间线雏形，但状态表达还可以更强，缺少明确的阶段提示和等待预期。

### 需求说明

- 订单详情页时间线至少展示：
  - 订单已创建
  - 待付款
  - 已上传截图
  - 商家审核中
  - 已确认到账
  - 配送中
  - 已完成
- 用户提交截图后，页面应立即显示强反馈：
  - `付款截图已提交，等待商家确认`
- 如果支付状态为 `FAILED`，需明确告知用户重新上传
- 页面预留联系商家入口位置

### 开发建议

- 以 [order-detail.js](/Users/liuxu/Desktop/codex/takeaway-app/miniprogram-user/pages/order-detail/order-detail.js) 为主改造
- 当前后端已有足够状态字段，优先先做前端状态表达
- 联系商家入口如果当前没有真实联系方式，可先预留 UI 和静态文案

### 验收标准

- 用户能通过订单详情页清楚知道订单当前卡在哪一步
- 截图提交后有明显成功反馈
- 截图被退回后，用户能明确知道需要重传

## 6.5 用户端：登录页反馈增强

### 目标

减少“登录只是被迫授权”的感觉，明确告诉用户登录后能做什么。

### 需求说明

- 登录页标题明确改为结果导向文案
- 登录按钮点击后显示明确 loading
- 登录成功后显示 toast 或状态提示
- 登录后按来源页返回

### 开发建议

- 以现有登录页实现为基础做文案和反馈优化
- 不修改微信登录接口设计

### 验收标准

- 登录时有清晰进行中状态
- 登录成功后有清晰完成反馈
- 回跳逻辑不丢失

## 6.6 商家端：首页工作台继续收敛

### 目标

让首页真正回答商家两个问题：

- 今天先处理什么
- 当前经营情况怎么样

### 当前问题

当前首页已有数据卡片，但还可以更明确地区分待处理事项和经营数据。

### 需求说明

- 首页首屏优先展示待办：
  - 待审核付款
  - 待配送订单
  - 未读消息
- 次级展示经营数据：
  - 今日订单数
  - 今日销售额
  - 新增用户数
  - 启用规则数
- 快捷入口排序固定为：
  - 订单管理
  - 规则管理
  - 商品管理
  - 用户中心
  - 店铺设置

### 开发建议

- 以 [merchant index.js](/Users/liuxu/Desktop/codex/takeaway-app/miniprogram-merchant/pages/index/index.js) 和对应样式为主
- 优先做信息层级优化，不新增复杂接口

### 验收标准

- 商家首屏能一眼看出先处理什么
- 重要数据优先级稳定

## 6.7 商家端：商品管理增强筛选和效率

### 目标

降低商家维护菜单的操作成本。

### 需求说明

商品管理页新增：

- 关键词搜索
- 按分类筛选
- 按上下架筛选
- 按午餐 / 晚餐可点筛选

后续可选增强：

- 商品复制
- 批量上下架

### 开发建议

- 先做前端筛选即可，优先使用当前商品列表接口
- 数据量如果后续增大，再考虑后端筛选参数

### 验收标准

- 商家能快速定位目标商品
- 不需要手动滚长列表找商品

## 6.8 商家端：规则管理增强风险提示和预览

### 目标

让商家改规则时更有把握，减少“改了但不知道会发生什么”的风险。

### 当前问题

当前已有组合预览接口，但规则改动前后的影响感知仍然偏弱。

### 需求说明

- 规则页继续强化以下信息：
  - 当前启用规则列表
  - 某组荤素数量能否结算
  - 恢复默认规则前的二次确认
- 规则保存后应有明确反馈
- 如果规则导致某些常见组合无法结算，应给出明显提示

### 开发建议

- 继续复用已有 `/api/merchant/combo-rules/preview`
- 前端增加高频组合快捷预览，例如：
  - 1 荤 1 素
  - 2 荤 1 素
  - 1 荤 2 素

### 验收标准

- 商家能在规则页快速判断常见组合是否还能卖
- 错误配置风险下降

## 7. 技术改动建议

本轮建议遵循以下原则：

- 优先改前端展示和交互，不优先大改后端模型
- 优先复用现有接口和状态字段
- 只有当前端无法满足时，再增加小型补充接口
- 新增接口应继续保持当前 REST 风格

后端如需补充，优先考虑：

- 订单详情返回更明确的时间线字段
- 店铺信息增加联系商家字段
- 商品列表接口支持筛选参数

## 8. 开发排期建议

建议按两期推进。

### 第一阶段

- 游客可浏览菜单
- 套餐反馈增强
- 购物车页核对化
- 订单详情时间线增强
- 商家首页工作台收敛

### 第二阶段

- 商品管理筛选
- 规则页高频组合预览增强
- 登录反馈增强
- 联系商家入口
- 商品复制 / 批量上下架

## 9. 风险和注意事项

- 游客浏览改造要注意不要破坏当前已登录用户主流程
- 前端套餐预览必须与后端结算保持一致，不能出现展示可下单但后端拒单
- 商家规则改动会直接影响前台结算，相关页面必须保留明确提示
- 当前仓库仍有未提交文档整理改动，提交开发代码时要避免混入无关文件

## 10. 交付定义

本轮完成的标志不是“页面更多”，而是：

- 用户首次进入时更容易理解产品
- 用户选菜时更容易理解价格和套餐
- 用户付款后更清楚订单进度
- 商家每天打开后台时更清楚优先处理什么

如果上述四点没有显著改善，就不算真正完成本轮迭代。
