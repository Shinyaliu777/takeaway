## 1. 目标

本轮迭代同时吸收产品经理和系统架构师两类意见：

- 产品侧优先解决转化、理解成本、支付信任感、商家操作效率
- 架构侧优先控制复杂度、减少单点耦合、避免大规模重构

本轮原则：

- 先优化主链路，不扩新业务
- 先做低风险高收益改动，不做重型重构
- 前端体验增强优先复用现有接口和数据结构
- 后端只做必要的结构整理和小补充

## 2. 总体优先级

### P0

- 用户端允许游客浏览菜单
- 用户端强化套餐结算反馈和付款进度反馈
- 商家端首页明确待处理优先级

### P1

- 用户端登录反馈增强
- 商家端商品管理筛选增强
- 商家端规则管理预览增强
- 后端订单 / 支付逻辑做最小必要整理

### P2

- 联系商家入口
- 商品复制 / 批量操作
- 更系统的后端模块化拆分
- 正式 migration 体系

## 3. 子线程分工

## 3.1 子线程 A：用户端体验链路

负责人范围：

- `miniprogram-user/`

目标：

- 降低首次进入门槛
- 减少套餐理解成本
- 强化付款截图流程透明度

任务：

- 菜单页改为游客可浏览
- 加菜 / 去结算 / 地址 / 上传截图改为动作级登录拦截
- 底部结算信息增强
- 购物车页进一步核对化
- 订单详情页状态反馈增强
- 登录页反馈优化

不做：

- 改后端协议
- 重写结算逻辑

## 3.2 子线程 B：商家端工作台和管理效率

负责人范围：

- `miniprogram-merchant/`

目标：

- 首页更像工作台
- 商品管理效率更高
- 规则改动风险更低

任务：

- 首页强化待审核 / 待配送 / 未读消息优先级
- 商品管理新增本地搜索和筛选
- 规则页增加高频组合预览和更明确反馈

不做：

- 引入复杂图表系统
- 重写商家信息架构

## 3.3 子线程 C：后端最小结构治理

负责人范围：

- `backend/`

目标：

- 在不大改接口的前提下，降低后续迭代风险

任务：

- 抽离安全且边界明确的订单 / 支付状态流转辅助逻辑
- 保持现有 API 兼容
- 只补充本轮前端确实需要的小能力

不做：

- 微服务拆分
- 引入完整新架构
- 大规模 schema 改造

## 4. 合并原则

- 以用户主链路可用性为第一验收标准
- 前端展示逻辑不得偏离后端最终结算结果
- 商家端优化优先保持当前接口兼容
- 后端重构不得改变现有接口响应结构，除非前后端同步确认

## 5. 验收顺序

1. 用户端主链路
2. 商家端工作台和管理页
3. 后端结构整理
4. 联合回归

## 6. 本轮完成标志

满足以下条件，才算本轮真正完成：

- 用户未登录也能浏览菜单
- 用户能清楚理解当前套餐是否可结算
- 用户能清楚知道付款截图提交后的进度
- 商家打开首页能一眼看到今天优先处理什么
- 后端没有因为本轮迭代进一步加重耦合
